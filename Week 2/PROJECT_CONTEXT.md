# Project Context (Week 2)

## 1) Project Purpose

This project supports a class assignment and personal learning in UX research.

Primary goals:
- Demonstrate effective processing of a data file.
- Explain important parts of the code clearly for a non-engineering audience.
- Keep the work easy to run and easy to discuss in class.

## 2) Practitioner Profile

Project owner perspective:
- Role: UX Research
- Background: HCD practitioner, not a software engineer
- Preferred implementation style: beginner-friendly with some structure

Implication for coding work:
- Favor clarity over advanced patterns.
- Use clear naming and step-by-step logic.
- Teach through code and documentation, not just output.

## 3) Audience and Success Criteria

Primary evaluation priority:
- Clear explanation of the code

Likely audience:
- Instructor/TA and course peers
- Future self reviewing project decisions

What "good" looks like:
- A reader can quickly understand what the script does, why it does it, and how to run it.
- Key logic is visible and explained.
- Output is understandable without deep programming knowledge.

## 4) Technical Preferences and Constraints

Environment:
- Run with `python3` in macOS terminal.
- Keep scripts runnable from the `Week 2` directory.

Dependencies:
- Use Python standard library only unless explicitly approved otherwise.

Data handling:
- Do not modify original source CSV files unless explicitly requested.
- Prefer safe and explicit file handling.

Code structure:
- Keep single-file script for now unless complexity clearly justifies splitting.
- Use beginner-friendly structure: small functions + clear sections.
- Include teaching-style comments throughout, especially around key logic.

Error handling:
- Use both:
  - friendly user-facing error messages
  - optional debug details (traceback) when needed for troubleshooting

## 5) Output and Documentation Expectations

Current output preference:
- Script should run and produce understandable results directly.
- Browser output is optional in future iterations.

Documentation requirements:
- Create a separate script explanation document in `Week 2`.
- Explanation format: medium detail (6-8 sections with examples where helpful).
- Tone: adaptive mix of professional and teaching style, depending on topic.

## 6) AI Assistance Guidance (for future edits)

When generating or revising code in this project:
- Explain decisions briefly in plain language.
- Prefer explicit, readable steps over compact one-liners.
- Preserve working behavior before adding new features.
- Include exact run commands when sharing changes.
- Highlight the code sections that are most important to understand.

When generating or revising docs:
- Keep explanations concise but instructional.
- Use section-based structure and practical examples.
- Tie technical behavior back to UX research relevance.

## 7) Class Presentation Talking Points

When presenting this work:
- Method: Explain the data-processing flow from CSV input to summary output.
- Code choices: Explain why simple, readable Python was chosen.
- Reliability: Note path handling and basic error messaging choices.
- Insight framing: Connect output metrics (for example, response length patterns) to possible UX research interpretation.
- Reflection: Mention what could be improved next (for example, HTML output/reporting) while keeping the current version intentionally beginner-friendly.

## 8) Scope Note

This context file is specific to Week 2 deliverables and may evolve as course requirements change.
