## C5 — Data Analysis with Pandas

The notebook answers three specific analytical questions on a 20,985-row, 11-column dataset of YouTube livestream chat messages. Q1 uses `groupby("offset_min_bin_5").size()` and a merge with `groupby().agg(first_ts="min", last_ts="max")` to compute messages per minute in 5-minute windows and locate the top burst windows in UTC wall-clock time; the busiest window reaches 47.8 messages per minute against a median of 15.8. Q2 uses `value_counts()` on `author_channel_id` followed by a cumulative-sum operation on the ranked series to find that the busiest 10% of accounts contribute 65.4% of all messages, with cumulative shares of 81.7% at the top quartile and 91.9% at the top half — a clear heavy-tailed distribution. Q3 combines `groupby().agg()` with multiple named aggregations (count, `nunique`, and mean-of-booleans for the global-top-10% share), a median split using `np.where()`, and a second `groupby()` to compare high- and low-activity bins, finding that busier windows average 64.5 unique authors versus 22.6 in quieter windows while the heavy-poster share is 65.4% in busy bins versus 68.2% in quiet ones, meaning louder periods involve broader participation rather than fewer dominant accounts. Each question has a written interpretation cell that ties the specific pandas output back to a chat-behavior reading without overclaiming causation.

## C6 — Data Visualization

Section 4 publishes three Plotly Express charts in a Jupyter notebook on GitHub, each with code, output, and markdown justification cells. I chose chart types deliberately: Chart 1 uses a horizontal bar (`px.bar(..., orientation="h")`) limited to the twelve loudest 5-minute windows because the time-slice labels are long and a vertical bar would render them illegibly, and because showing only the top windows answers "when should a producer look?" more directly than a wiggly full-timeline line plot; Chart 2 uses a two-bar split because the analytical claim is a single binary contrast (top 10% of accounts versus everyone else) and any third visual element would add noise without information; Chart 3 deliberately splits into two side-by-side bar charts rather than a single dual-axis chart because the two metrics (counts of unique authors and percentages of lines) have incompatible scales, and a shared y-axis would mislead the reader. Every chart title states the finding rather than the variable ("When was chat fastest?", "Who writes most of the chat on this stream?"), every axis is explicitly labeled with units and computation method, and each chart carries a Plotly `annotations` footnote clarifying the underlying bin or rate calculation so the visualization is interpretable without re-reading the code.

## C7 — Critical Evaluation and Professional Judgment

I overrode AI-generated output at multiple points in this project. The clearest example: when I asked Cursor to refine my analysis code, it proposed verifying that outputs were unchanged by hashing string representations and comparing across runs. I rejected this because Python's `hash()` of strings is randomized per process by default (PEP 456), making the check unreliable as a regression test; I redirected the tool to do semantic value comparison against the previously reported numbers (Q2 share of 0.6539957345376992 and the Q1 burst offsets) instead. A second example: Cursor's initial chart code labeled Chart 1's rate axis ambiguously, and I would not have shown that to a stakeholder without correcting it; I had Cursor add an explicit annotation stating the horizontal axis is messages per minute calculated as bin count divided by five. The largest judgment call in the project was the Q3 pivot itself: my original Question 3 compared Super Chat frequency across high- and low-activity periods, but the export contained zero paid-message events. Rather than report a null finding as if it were meaningful or fabricate a workaround, I replaced Q3 with a participation-shape analysis the data could actually support and documented the pivot rationale explicitly in Section 1. The conclusion bounds what the analysis can and cannot claim: no in-game causal attribution because I have only chat timestamps and no event log, no paid-engagement analysis because Super Chats are absent, and findings limited to the official LEC broadcast export rather than the co-stream venues where chat density may actually concentrate.
